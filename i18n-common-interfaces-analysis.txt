1) i18n setup summary

- File analyzed: frontend/src/lib/i18n.ts
- Library stack: i18next + react-i18next + i18next-browser-languagedetector
- Translation resources are loaded statically at startup from:
  - frontend/messages/en.json
  - frontend/messages/fr.json
  - frontend/messages/ar.json
- Resource registration pattern:
  - resources.en.translation = en
  - resources.fr.translation = fr
  - resources.ar.translation = ar
- Language defaults:
  - lng: "en"
  - fallbackLng: "en"
- Detection strategy:
  - order: localStorage -> cookie -> navigator
  - caches: localStorage
- Interpolation:
  - escapeValue: false
  - custom token delimiters: { and }

Namespace strategy confirmation
- Single namespace strategy is active:
  - ns: ["translation"]
  - defaultNS: "translation"
- Usage pattern aligns with dot-path keys:
  - t("admin.dashboard.title")
  - t("landing.hero.heading")
  - t("common.actions.submit")

2) JSON structure explanation (frontend/messages/en.json)

- The file is a nested domain-based key tree (object-of-objects), not a flat key map.
- Dot notation in code resolves nested paths inside the single "translation" namespace.
- Top-level domains currently present include:
  - auth
  - profile
  - common
  - dashboard
  - landing
  - layout
  - events
  - admin
  - organizer
  - enterprise
  - visitor
- Key style pattern:
  - Domain -> Subdomain -> Field
  - Examples:
    - auth.login.title
    - common.navigation.home
    - landing.hero.title
    - events.detail.joinNow
- Variable interpolation uses braces in message strings:
  - Example style: "Joined {year}", "Showing {from} to {to}"
